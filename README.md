# OneBusiness — Дашборд расхождений (ArmSoft / TaxService / выгрузка Артёма)

Mobile-first веб-дашборд для отслеживания расхождений между:

- реестром клиентов OneBusiness (`ob_accounting_companies`),
- налоговым кабинетом **TaxService** (`v_tax_accounts`, данные Артёма),
- **ArmSoft** (`v_armsoft_companies`, данные Артёма),
- списком-выгрузкой **Артёма** (`artem_companies`),
- отчётами бухгалтеров с утренних встреч (`accountant_daily_comments`).

Backend — существующий Supabase-проект **OB Artyom** (`rbtvbsbcycdlwmrzjwun`).
Чистый HTML/CSS/JS (без фреймворков); Supabase JS v2 лежит локально в
`public/js/vendor/`. Vite используется только как dev-сервер и для сборки.

2. **Выгрузки** — история выгрузок Артёма за всё время: график по датам
   (высота столбца = сколько модулей-парсеров отработало в этот день),
   **реальный объём выгрузки по всему проекту OB Artyom** (точное число строк
   по каждому набору данных — журналы, счета, операции и т.д., а не только

   нет в TaxService» (с поиском; неточные совпадения «≈» не исключаются,

3. **Дельта** — каждый показатель кликабелен: открывается список конкретных

   датами и источником.
4. **Проверка с Эмилией** — для каждого расхождения: 7 статусов подтверждения,
   комментарий, ответственный, приоритет. Всё сохраняется в Supabase
   (`delta_items`), не теряется после обновления страницы.
5. **ТЗ Артёму** — автоматически собирает расхождения со статусом
   «Проблема выгрузки Артёма» и генерирует готовое русскоязычное сообщение
   для копирования.
6. **Встречи** — сравнение того, что бухгалтеры отчитали на утренних встречах,
   с выгрузкой Артёма, TaxService и ArmSoft (включая неточные совпадения «≈»).
7. **Синхронизация** — сверка ежедневных задач бухгалтеров с выгрузкой Артёма
   (см. отдельный раздел ниже): статус графика выгрузки (4 состояния), отчёт по
   процессу с 2 ключевыми показателями, по каждому бухгалтеру — задачи, не
   отражённые в выгрузке, список проблем и поле самопроверки бухгалтера.

## Синхронизация задач бухгалтеров с выгрузкой Артёма

Выгрузка Артёма — единственный источник **полной** бухгалтерской информации,
поэтому все ежедневные задачи бухгалтеров должны с ней сверяться. Раздел
«Синхронизация» отвечает на два ключевых вопроса (отчёт по процессу):

1. **Выгрузил Артём или нет** — статус графика выгрузки: `⏳ Ожидаем выгрузку`,
   `✅ Выгрузил`, `⚠️ Просрочено`, `⚫ Нет данных`. Считается SQL-функцией
   `public.artyom_export_schedule_status()` по времени последнего прогона
   парсеров и настройкам графика (см. ниже) — единый источник статуса и для
   дашборда, и для Telegram-отбивки.
2. **Соответствует ли выгрузка действительности** — доля задач бухгалтеров,
   реально отражённых в выгрузке (`в выгрузке / ожидалось`).

**Что считается задачей.** Ежедневные задачи берутся из `accounting_activities`
(счётчики: счета/инвойсы, сдача отчётов, подача заявлений, изменение остатков)
и из `accountant_daily_comments.unaccounted_work` (что бухгалтер сам отметил как
«не отражено»). Каждая задача сверяется с выгрузкой Артёма по системе задачи
(налоговая → `v_tax_accounts`, ArmSoft → `v_armsoft_companies`) тем же движком
сопоставления, что и дельта. Статусы задачи: `in_export` (отражена),
`missing_in_export` (нет в выгрузке), `not_expected` (Артём её в принципе
не видит — по config), `problem`.

**Задачи, которые Артём не видит** (согласовать с Эмилией и Лилит) — в
`config.js → TASK_SYNC.tasksNotInExport`: список ключей типов задач
(`taskTypeKeys`) и текстовых паттернов (`patterns`: устные согласования,
консультации, ручные корректировки прошлых периодов, обучение, внутренняя
работа и т.д.). Держится в конфиге, а не в коде.

**Список проблем** (`sync_problems`): расхождения (`coverage_gap`), ошибки
названия/формата (`name_mismatch`), не отражённая работа (`unaccounted_work`),
статус графика (`schedule`). Статус разбора (`open/ack/resolved`) сохраняется.

**Самопроверка бухгалтера (следующий этап).** Модель данных уже заложена:
в `accountant_task_sync` есть поля `accountant_response_status`
(`pending / confirmed_ok / reported_missing / disputed`), `accountant_response`,
`accountant_confirmed`, `accountant_checked_at`. В UI по каждой не отражённой
задаче бухгалтер выбирает статус и оставляет комментарий — это заготовка
процесса, где каждый бухгалтер сам подтверждает корректность своих данных
вместо разовой сверки аналитиком.

### График выгрузки и Telegram-отбивка

Настройки графика хранятся **в БД** (`public.app_config`, канон для серверной
проверки) и продублированы в `config.js → TASK_SYNC.exportSchedule` (для UI):

| app_config key | По умолчанию | Смысл |
|---|---|---|
| `export.frequency` | `daily` | как часто ждём выгрузку |
| `export.expected_hour_yerevan` | `2` | к какому часу (Ереван) ждём свежую выгрузку |
| `export.grace_hours` | `12` | льготный период до статуса «Просрочено» |

Edge-функция `supabase/functions/artyom-export-notice` дёргает
`v_artyom_export_status` и шлёт сообщение с одним из 4 статусов в Telegram.
Секреты — **только в переменных окружения** (Supabase → Edge Functions →
Secrets), в репозиторий не попадают:

```
TELEGRAM_BOT_TOKEN   # токен бота
TELEGRAM_CHAT_ID     # чат/канал назначения
NOTICE_SECRET        # (опц.) общий секрет; тогда запрос обязан прислать
                     # заголовок x-notice-secret с тем же значением
```

Без `TELEGRAM_*` функция работает в режиме **dry run** и возвращает готовое
сообщение в теле ответа (удобно проверить без секретов). Ежедневный запуск —
через Scheduled Functions / cron на нужный час (после дедлайна выгрузки).

## Модель данных и бизнес-логика (настраивается в `public/js/config.js` → `RULES`)

**Что такое «выгрузка Артёма».** Весь проект OB Artyom — это результат его
выгрузки. Его парсеры (`armsoft_db.parser_modules`) переносят данные из
налогового кабинета и ArmSoft в таблицы `armsoft_db`. Поэтому «выгрузка
Артёма» = фактически разобранные данные:
`TaxService → v_tax_accounts`, `ArmSoft → v_armsoft_companies`.
**Источник истины «наш активный клиент»** — реестр `ob_accounting_companies`.

Главная цель дельты: сколько активных клиентов OB ещё **не попали** в выгрузку
Артёма. После каждой новой выгрузки этот разрыв должен уменьшаться — это и видно
в сводной таблице по дням.

> ⚠️ Таблица `artem_companies` (15 строк с тестовыми ИНН `AM00…`) — это
> демонстрационные данные, в расчёте дельты **не используется**.

В реестре клиентов **нет собственного поля ՀՎՀՀ** — оно восстанавливается
через связь `tax_account_id → v_tax_accounts.tin` или совпадение названия
(поэтому у клиента, которого нет в TaxService, ՀՎՀՀ показывается пустым).

- **Правило TaxService** (`expectedInTaxService`): активный клиент обязан быть
  в налоговом кабинете. Дельта = активные клиенты OB, которых нет в налоговой
  выгрузке. ՀՎՀՀ показывается там, где известен, и доступен как фильтр. В коде
  на одной закомментированной строке лежит строгий вариант «активен **и** есть
  ՀՎՀՀ» — раскомментируйте, если нужно.
- **Правило ArmSoft** (`expectedInArmsoft`): клиент обязан быть в ArmSoft, если
  у него заполнена ArmSoft-привязка (`armsoft_company_id` — признак «заведена
  учётка ArmSoft»). Логика **отдельная** от TaxService и легко меняется.

Типы расхождений (любой можно отключить в `RULES.ENABLED_ISSUE_TYPES`):

| Тип | Смысл | Направление |
|---|---|---|
| `missing_taxservice` | активный клиент OB → нет в налоговой выгрузке | прямое (в дельте) |
| `missing_armsoft` | клиент OB с ArmSoft-привязкой → нет в ArmSoft-выгрузке | прямое (в дельте) |
| `tax_not_in_ob` | есть в налоговой выгрузке Артёма → нет в реестре OB | обратное |
| `armsoft_not_in_ob` | есть в ArmSoft-выгрузке Артёма → нет в реестре OB | обратное |
| `meeting_not_in_export` | упомянута на встрече → нет в выгрузке Артёма | встречи |

«Общая дельта» в дневной таблице и карточка «Дельта сегодня» = только **прямые**
типы (`missing_taxservice + missing_armsoft`) — именно они уменьшаются после
выгрузок Артёма. Обратные типы остаются кликабельными в разделе «Дельта», но не
раздувают ежедневный итог.

**Сопоставление**: сначала точное по ՀՎՀՀ → затем по нормализованному
названию (lowercase, без кавычек/пунктуации, без ООО/LLC/ՍՊԸ/ԱՁ/ИП, схлопнутые
пробелы) → затем нечёткое (транслит кириллица↔латиница). Нечёткие совпадения
помечаются «≈» и **не** считаются точными.

## Данные в Supabase (созданы этой задачей, миграция аддитивная)

- `daily_delta_snapshots` — один снимок в день (уникальна `snapshot_date`);
  `delta_change` = сравнение с предыдущим днём.
- `delta_items` — живой реестр расхождений. Уникальный `issue_key`
  (тип + ՀՎՀՀ/название) сохраняет статусы проверки при пересчётах;
  исчезнувшие расхождения закрываются `resolved_at`, вернувшиеся —
  переоткрываются.
- `artyom_tz_items` — задачи Артёму (автозаполнение при статусе
  «Проблема выгрузки Артёма»).
- `v_artyom_export_meta` — время последнего прогона парсеров Артёма
  и количества записей.
- `v_artyom_export_dates` — история выгрузок по датам (для графика в разделе
  «Выгрузки»): дата, сколько модулей отработало, время последнего прогона.
- `v_artyom_export_volume` — реальный объём выгрузки по **всему проекту**:
  точное число строк по каждому набору данных `armsoft_db` (ArmSoft +
  TaxService). Раньше «объём выгрузки» считался только по двум справочникам
  компаний (≈1304 строки), что кратно занижало картину — на деле парсеры
  загрузили ~1.6 млн строк. Считает SECURITY DEFINER-функция
  `public.artyom_export_volume()` (у `anon` нет прямого доступа к `armsoft_db`).
- `exists_in_ob_registry` — колонка добавлена миграцией
  `delta_items_add_ob_registry_flag` (аддитивно): в реестре OB или нет.

**Восстановленные public-view** (миграция `restore_public_export_views`,
`sql/2026-07-20_restore_public_export_views.sql`). PostgREST читает только схему
`public`, а базовые таблицы Артёма лежат в `armsoft_db`. Когда парсеры Артёма
пересоздают `armsoft_db`, зависимые public-view удаляются каскадом — так пропали
`v_armsoft_companies`, `v_artyom_export_meta`, `v_artyom_export_dates`, из-за чего
дашборд падал с `Could not find the table 'public.…'`. Миграция их пересоздаёт
(и не выставляет учётные данные: из `armsoft_companies` не отдаются
`local_token/login_raw_data/service_url`, из `tax_accounts` — `username/password`).
Если ошибка «Could not find the table» повторится после пересоздания `armsoft_db`
— повторно примените эту миграцию (все `CREATE OR REPLACE`, идемпотентно).

**Реальная работа бухгалтеров** (миграция `accountant_work_feed`,
`sql/2026-07-20_accountant_work_feed.sql`). Страница «Бухгалтеры» раньше брала
«задачи» только из `accounting_activities` (76 компаний под общим «OB Accounting»
с короткими ключами, не совпадавшими с реестром) — поэтому у всех стояло
«— нет отчёта». При этом парсеры Артёма уже разобрали фактическую работу по
каждой компании (~46 000 выданных счетов, ~47 000 полученных, ~5 900 сданных
налоговых отчётов, налоговые э-счета) — эти данные не выводились. Миграция
аддитивно выставляет их в `public`:

- `v_ob_arm_activity` — сводка работы по компании ArmSoft (`company_id`):
  выставлено/получено счетов, дата последнего документа.
- `v_ob_tax_activity` — сводка работы по налоговому кабинету (по ИНН/`tin`):
  сдано отчётов, выдано/получено налоговых счетов, дата последней активности.
- `ob_company_task_feed(armsoft_company_id, tin, limit)` — SECURITY DEFINER RPC:
  единый список КОНКРЕТНЫХ задач одной компании (каждый выставленный счёт,
  каждый сданный отчёт — с датой, партнёром, суммой, периодом, статусом), для
  drill-down «Показать задачи» на карточке компании.

Мост к клиентам OB строит фронтенд в `normalize.js` (сопоставление уже есть):
`tin` берётся из налогового совпадения, `company_id` — из ArmSoft-совпадения.
Страница «Бухгалтеры» переписана: KPI по реальной работе, карточка на каждую
компанию (где существует + сводка работы) и раскрываемый список задач.

**Синхронизация задач** (миграция `accountant_task_sync`,
`sql/2026-07-20_accountant_task_sync.sql`):

- `public.artyom_export_schedule_status()` + view `v_artyom_export_status` —
  статус графика выгрузки (4 состояния), считает по `armsoft_db.parser_modules`
  и `app_config` (SECURITY DEFINER, как остальные view над `armsoft_db`).
- `public.accountant_task_sync` — реестр сверки задач + модель самопроверки
  бухгалтером; `task_key` (бухгалтер|компания|система|тип) сохраняет ответы
  бухгалтера при пересчётах.
- `public.sync_problems` — список проблем сверки (со статусом разбора).
- `app_config` — сид ключей графика выгрузки (`export.*`).

Пересчёт выполняется кнопкой «⟳ Пересчитать» (и автоматически один раз в день
при открытии). Существующие таблицы проекта не изменялись.

## Открытые вопросы для Эмилии и Лилит

- **Список задач, которые Артём не видит** (`config.js →
  TASK_SYNC.tasksNotInExport`): текущий список паттернов — черновой. Нужно
  пройтись по реальным ежедневным задачам и утвердить финальный перечень типов
  работ, которые в принципе не попадают в выгрузку.
- **Формат ответа бухгалтера** о неучтённых задачах: сейчас заложены статусы
  `confirmed_ok / reported_missing / disputed` + свободный комментарий.
  Согласовать, достаточно ли этого или нужны структурированные поля.
- **График выгрузки**: подтвердить фактическое расписание Артёма
  (`export.expected_hour_yerevan`, `export.grace_hours`, частота) и список
  обязательных модулей (`TASK_SYNC.exportSchedule.requiredModules`).
- **Глубина сверки**: сейчас задача считается «отражённой», если компания есть
  в выгрузке соответствующей системы. Нужно решить, проверять ли соответствие
  по периодам/объёмам (конкретные счета/отчёты), а не только присутствие компании.

## Безопасность

Инструмент внутренний: как и остальные таблицы проекта, новые таблицы доступны
anon-ключу, но у новых таблиц RLS **включён** с явными политиками — при
необходимости доступ легко ужесточить (например, только `authenticated`).
⚠ У пяти старых таблиц проекта (`ob_accounting_companies`,
`accountant_daily_comments`, `artem_companies`, `accounting_activities`,
`document_records`) RLS выключен — это существующее состояние проекта,
данная задача его не меняла.
